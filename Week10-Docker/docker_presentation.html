Why we need more than renv

You pin every package, push to GitHub, and your collaborator still cannot run your code. Why?

  • Different R or Python version — packages no longer compile
  • Missing system libraries like libxml2 or pandoc
  • Different operating system — subtle changes in results
  • renv.lock cannot fix any of these

The reproducibility stack

  • Every project sits on layers
  • Git versions only the top layer
  • renv locks only the package layer
  • Docker locks everything below too

So what is Docker?

Docker wraps your entire working environment — OS, runtime, libraries, code — into one sealed package that runs identically anywhere.

  • Solves the "works on my machine" problem
  • Lighter than a virtual machine
  • Standard in modern research and industry
One-line summary

Docker is renv for the whole operating system.

Three words to remember

The kitchen analogy

  • Dockerfile = the recipe (a text file)
  • Image = the meal, cooked and vacuum-sealed
  • Container = the meal being eaten
Why the separation matters

The recipe is written once. The meal is cooked once. But the meal can be reheated and served many times — each serving is independent.

One image, many containers

  • Image = cookie cutter
  • Container = one cookie
  • Each container is isolated
  • Create, destroy, repeat — cheaply

What happens when you run Docker

A typical Docker workflow has four moments. Here is what actually happens at each one.

  • Write the Dockerfile
  • Build it into an image
  • Run the image as a container
  • Exit — container disappears, image stays

The four-step timeline

Your computer stays clean

  • Nothing is installed on your Mac directly
  • Python, Linux, packages — all live inside the image
  • Delete the image → your computer is back to normal
  • No version conflicts with other projects
Why this matters

You can have five different Python versions running on the same laptop, each in its own image. None of them touch your system Python.

How someone else uses your work

Docker images live on whichever computer built them. For someone else to use one, something has to travel.

  • Two ways this can happen
  • Both are common in practice
  • Your choice depends on who you are sharing with

Two ways to share

Which way should you use?

  • Way 1 (Dockerfile via Git) — standard for research
  • Small, human-readable, lives with your code
  • Works for 90% of academic projects
  • Way 2 (Docker Hub) — for fast distribution
  • Published papers, teaching materials, production apps
  • Skip the build time for anyone who clones
Rule of thumb

For your research: ship the Dockerfile. That's it.

What we covered

  • The problem — reproducibility goes deeper than packages
  • The concept — Docker locks the whole stack
  • The three words — Dockerfile, image, container
  • The workflow — write, build, run, exit
  • Sharing — Dockerfile via Git, or image via Docker Hub
Still optional for your final project

renv or venv plus a clear README is enough. Docker is a tool you reach for when you need it.
