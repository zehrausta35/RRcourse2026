Docker Exercise: Your First Reproducible Container

Understand the three core Docker concepts — image, container, Dockerfile — by building and running a tiny Python script in an isolated environment.

💡 The whole exercise stays in the terminal. No text editor needed. This is itself good practice for reproducibility: every step is a command you can copy, paste, and re-run.

Why this exercise?

Reproducibility means: anyone, anywhere, on any machine, gets the same result. The classic problem — "it works on my machine" — happens because Python versions, package versions, and operating systems differ. Docker solves this by packaging your code together with its environment.

In this exercise, you will package a one-line Python script into a Docker image and run it. That's it. No web servers, no databases, no docker-compose. Just the absolute minimum.

Prerequisites

Install Docker Desktop from docker.com/products/docker-desktop and make sure it is running. Verify:

docker --version

You should see something like Docker version 27.x.x. If not, ask for help before continuing.

🪟 Windows users: the commands below use PowerShell syntax (which is what Windows Terminal opens by default). If you prefer Git Bash or WSL, use the macOS/Linux blocks instead.

Step 1.Create a working folder

mkdir hello-docker
cd hello-docker

Step 2.Create a tiny Python script

We will use a heredoc (cat > file << 'EOF' ... EOF) to write the file directly from the terminal. Everything between << 'EOF' and the closing EOF becomes the file content.

macOS / Linux / Git Bash / WSL

cat > hello.py << 'EOF'
import sys
print(f"Hello from Python {sys.version_info.major}.{sys.version_info.minor} inside a container!")
EOF

Windows PowerShell

@'
import sys
print(f"Hello from Python {sys.version_info.major}.{sys.version_info.minor} inside a container!")
'@ | Out-File -FilePath hello.py -Encoding utf8

Verify the file was created correctly:

cat hello.py        # macOS / Linux
type hello.py       # Windows PowerShell

Now try running the script on your own machine:

python hello.py

Note which Python version it prints. This is the version your computer happens to have. Different students will see different numbers — that is exactly the reproducibility problem we want to solve.

Step 3.Create the Dockerfile

Same idea, but now we write a Dockerfile (no extension, capital D).

macOS / Linux / Git Bash / WSL

cat > Dockerfile << 'EOF'
FROM python:3.11-slim
WORKDIR /app
COPY hello.py .
CMD ["python", "hello.py"]
EOF

Windows PowerShell

@'
FROM python:3.11-slim
WORKDIR /app
COPY hello.py .
CMD ["python", "hello.py"]
'@ | Out-File -FilePath Dockerfile -Encoding utf8

Four lines, four ideas:

  • FROM — the base image we build on top of (a minimal Linux with Python 3.11 already installed).
  • WORKDIR — where inside the container we will work.
  • COPY — bring the file from your computer into the image.
  • CMD — the default command to run when the container starts.

⚠️ Why a heredoc and not chained echo commands? A Dockerfile has multiple lines and special characters ([, ], "). Writing it line-by-line with echo "..." >> Dockerfile is fragile: shells may interpret the quotes or special characters. Heredocs treat the whole block as plain text — exactly what we want.

Step 4.Build the image

docker build -t hello-docker .

The -t hello-docker gives the image a name. The . at the end tells Docker to look for the Dockerfile in the current folder.

Watch the output. Docker downloads the Python base image (only the first time), then runs each instruction in your Dockerfile.

Step 5.Run the container

docker run --rm hello-docker

You should see:

Hello from Python 3.11 inside a container!

Notice: Even if your computer has Python 3.9, 3.10, or 3.13, the container always reports 3.11. That is reproducibility — the environment is fixed by the Dockerfile, not by your laptop.

The --rm flag tells Docker to delete the container after it finishes (otherwise stopped containers pile up).

Quick checks for understanding

Answer these in your own words. There is no need to submit anything — they are for you.

  1. What is the difference between an image and a container?
  2. Why did we pin the base image to python:3.11-slim instead of just python?
  3. If your collaborator clones your repository (with hello.py and Dockerfile), what commands do they need to run to reproduce your result exactly?

Bonus (optional, +5 minutes)

Modify hello.py so it also imports numpy. Overwrite the file with a new heredoc:

macOS / Linux

cat > hello.py << 'EOF'
import sys, numpy
print(f"Python {sys.version_info.major}.{sys.version_info.minor}, NumPy {numpy.__version__}")
EOF

Windows PowerShell

@'
import sys, numpy
print(f"Python {sys.version_info.major}.{sys.version_info.minor}, NumPy {numpy.__version__}")
'@ | Out-File -FilePath hello.py -Encoding utf8

Rebuild and run with docker build -t hello-docker . and then docker run --rm hello-docker. It will fail — numpy is not installed in the image.

Fix it by rewriting the Dockerfile to install numpy before copying the script:

macOS / Linux

cat > Dockerfile << 'EOF'
FROM python:3.11-slim
WORKDIR /app
RUN pip install numpy==1.26.4
COPY hello.py .
CMD ["python", "hello.py"]
EOF

Windows PowerShell

@'
FROM python:3.11-slim
WORKDIR /app
RUN pip install numpy==1.26.4
COPY hello.py .
CMD ["python", "hello.py"]
'@ | Out-File -FilePath Dockerfile -Encoding utf8

Rebuild and run:

docker build -t hello-docker .
docker run --rm hello-docker

Now the container has numpy pinned to a specific version — anyone who builds this image gets the exact same numpy, on the exact same Python, on the exact same Linux. That is full reproducibility in two files plus a few commands.

What to take away

A Dockerfile is a recipe written in plain text. Anyone with Docker can read it, build the same image, and run the same code in the same environment — regardless of their operating system or what's installed on their laptop. That is the core of reproducible computational research.

And because the entire workflow above lives in the terminal — no GUI, no text editor — you can paste it into a script, share it as a README, or include it verbatim in a paper's supplementary material. The instructions are the experiment.
