§ 01

Overview

Each group presents their final project in a 40-minute slot. The slot has four parts: a short talk, a live demo, Q&A from classmates and the instructor, and a short buffer for the next group to set up.

The distinctive part of this course's presentation format is the live clone-and-run demo. At some point during your slot, you will clone your own repository onto a fresh environment, project your screen, and run the whole project in front of the class.

§ 02

The 40-minute schedule

Two groups present per 90-minute lecture. Your slot is tight but workable. The timings below are hard caps — we will hold you to them, because the next group also needs their 40 minutes.

0:00–0:15
15 min

Presentation

Motivation, research question, data, methods, key results. Slides or a rendered Quarto/R Markdown document. Every group member speaks.

0:15–0:25
10 min

Live demo — fresh clone

You clone your own repo into an empty folder, set up the environment, and run the project end-to-end. Screen projected. Talk through what you're doing.

0:25–0:35
10 min

Q&A

Questions from classmates and the instructor. At least one question must come from a non-presenting group.

0:35–0:40
5 min

Transition

Next group plugs in, opens their files, sanity-checks the projector.

§ 03

Before the presentation — what to ship

Push your final repo to a public GitHub repository at least 24 hours before your slot. This gives classmates time to look at it beforehand and prepare questions.

What the repo must contain

  • A README.md at the root Project description, team members, how to set up the environment, how to run the project from start to finish, expected outputs, and how long it takes. Assume the reader has never seen your code.
  • All code Scripts, notebooks, Quarto/R Markdown files — everything needed to go from raw data to final output. No "email me for this part" gaps.
  • Data — or a script that fetches it Small datasets committed directly. Large ones fetched by a script (with a stable source), or clearly documented if they require access credentials.
  • A dependency manifest For R: renv.lock, or at minimum a sessionInfo() file listing package versions. For Python: requirements.txt, environment.yml, or pyproject.toml. Pinned versions preferred.
  • No absolute paths Nothing that looks like /Users/your-name/Documents/... or C:\Users\.... Use relative paths or the here package (R) / pathlib (Python).
  • A .gitignore So you're not committing .DS_Store, __pycache__, .Rhistory, rendered HTML outputs, virtual environments, or credentials.
  • Final rendered output committed So anyone can see what the project is supposed to produce, even if they don't run it.

README.md — minimum viable structure

Your README is the single most important file in the repository. A classmate should be able to open it and know exactly what to do next. Aim for something like:

# Project title

## Authors
Names, course group, year.

## What this project does
Two or three sentences. No jargon.

## Requirements
- R >= 4.3, or Python >= 3.10
- List of system-level dependencies if any

## Setup
git clone <repo-url>
cd <repo-name>

# R projects
Rscript -e 'renv::restore()'

# Python projects
python -m venv .venv
source .venv/bin/activate      # or .venv\Scripts\activate on Windows
pip install -r requirements.txt

## How to run
quarto render report.qmd
# or
python main.py

## Expected output
Describe what gets produced and roughly how long it takes.

## Repository structure
Short tree of the main folders.
On "best effort"

This course grades reproducibility on documentation quality, not purity of tooling. You are not required to use Docker. What we grade is: could a reasonable person, reading your README, run this project on their laptop without messaging you? If yes — you are in good shape.

§ 04

During the presentation — the live demo

The live demo is the 10 minutes after your talk. Here is exactly how it goes.

Setup — before you take the stage

  1. Empty folder ready. On your laptop, have an empty directory open in a terminal. Name it something like ~/demo/. Delete any previous clones.
  2. A fresh terminal session. No history, no leftover variables, no cached packages visible on screen.
  3. Font size cranked up. Terminal, editor, browser. The back row should be able to read it.
  4. WiFi confirmed working. You will be cloning and pulling packages.

On stage — the demo script

Narrate what you are doing. Not "now I'm typing git clone," but "the first thing any new user does is clone the repo — here it goes." Treat the audience like they are the new user.

# Step 1 — clone from scratch
git clone https://github.com/your-group/your-project.git
cd your-project

# Step 2 — show the structure
ls
cat README.md     # or open it in your editor

# Step 3 — restore the environment
# R example:
Rscript -e 'renv::restore()'

# Python example:
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

# Step 4 — run the project
quarto render report.qmd
# or whatever your main entry point is

# Step 5 — open the output
open report.html

What you should be talking about while it runs

Package installs take minutes. Do not stand in silence. Use that time to:

  • Walk through your folder structure
  • Point out the key scripts and what each one does
  • Show the renv.lock / requirements.txt and explain why versions are pinned
  • Highlight any particularly tricky reproducibility decisions you made
Common pitfalls to pre-empt

Hardcoded paths. Uncommitted data files. Untracked .env files with credentials. Platform-specific dependencies (something that only installs on macOS). System libraries you installed once and forgot (e.g. libxml2, pandoc). Git LFS files you never pushed. Check all of these before your slot.

§ 05

Roles within your groupOptional

You have 3–4 people. Assign clear roles before the day of presentation, not during. Here is a minimal division of labour that works well for 40-minute slots.

Role 01

The Presenter

  • Introduces the project and the team
  • Handles the motivation and research question
  • Owns the first half of the talk
Role 02

The Methodologist

  • Explains the methods and key results
  • Owns the second half of the talk
  • Handles methods questions in Q&A
Role 03

The Demo Driver

  • Runs the live clone-and-run
  • Has the fresh terminal and projector set up
  • Handles infrastructure questions in Q&A
Role 04

The Timekeeper & Backup

  • Watches the clock, signals the team at checkpoints
  • Backup driver if the demo machine dies
  • Fields Q&A questions and distributes them to the right teammate

If you have three people instead of four, roles 3 and 4 can be merged — but the timekeeping function still needs to exist, even if just as a watch alarm.

§ 06

When things break on stage

It will happen to at least a few groups this semester. The question is not whether it happens, but how you respond.

The good response

  1. Name what you see. "Okay, the install is failing — let me read the error." Out loud. The audience is with you.
  2. Fix or describe. If you can fix it in 60 seconds, fix it. If not, describe what should happen: "this is where the regression output would normally appear — you can see the expected version in report.html which is committed in the repo."
A failure you can diagnose is fine

If the demo breaks and you can say "this is because we forgot to pin the dplyr version and 1.1.4 changed the behaviour of summarise" — that is a strong answer. It shows you understand your project. A working demo with no insight is actually worse.

§ 07

How this is graded

The presentation grade is a single number out of 100, composed of the following weights. Reproducibility is the largest single component — this is a reproducible research course.

ComponentWhat we look forWeight
Talk quality Clear motivation, methods, results. Every member speaks. 20%
Repository & documentation README clarity, folder structure, pinned deps, no hardcoded paths. 30%
Live demo Successful clone-and-run, or honest diagnosis when things break. 25%
Methods & analysis Is the research question answered? Are the methods appropriate? 15%
Q&A handling Answering questions honestly, distributing them among teammates. 10%
Total 100%

Note that a broken demo you can explain scores higher than a perfect demo with no reflection. The goal of this course is not that you never make mistakes, but that you can see them, name them, and teach the next person to avoid them.
