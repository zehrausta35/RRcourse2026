§ 01

What the project is for

The point is not to do the most complicated analysis you can imagine. The point is to produce a piece of research that someone else can actually run on their own computer, reach the same results, and understand what you did.

Everything in this course — version control, Quarto, coding principles, environments — is building toward one outcome: a research artefact that is genuinely reproducible. The final project is where you put it all together.

Your project should demonstrate:

  • A clear research question — something you set out to investigate, answer, model, or understand.
  • Real data — a dataset relevant to your question, either public or scraped/collected by you.
  • A documented analysis — code that produces results, with enough explanation that a classmate reading the repo understands what each part does.
  • Reproducibility — someone cloning your repo should be able to run it and get the same results.

Ambition matters less than execution. A small, clean, well-documented project is better than a sprawling, half-finished one.

§ 02

Document your project plan in the README

Before you start coding, agree as a group on what your project is about and write it down — not in an email, not in a shared Google Doc, but directly in the README.md file at the root of your Git repository.

This is more than a formality. Writing your plan forces the group to align on what you're actually trying to do, and keeping it in the README means future-you (and anyone reading your repository) can see the intent behind the code.

What your initial README should contain

Initial README Structure Before you start coding
Project title
A short, descriptive name for your project.
Group members
Full names of all 3–4 group members.
Research question
One or two sentences. What are you trying to find out, model, compare, or explain?
Data source
Where is the data coming from? A public dataset (link it), scraping (which site), an API (which one), an open repository (Kaggle, OECD, Eurostat, Zenodo)?
Planned approach
Roughly what methods or tools you plan to use. A few sentences. This is not a commitment — you can change course later.
Language / tools
R, Python, or both? Any specific packages or frameworks you already know you will need?
Motivation
One or two sentences on why your group finds this interesting.

As the project develops, your README will grow. What starts as a project plan becomes setup instructions, run instructions, and documentation of what the project produces. See section 5 for what the full README should contain by the end.

§ 03

Choosing a good topic

Because you choose your own topic, the temptation is to either pick something you already know backwards (too easy, nothing learned) or something wildly ambitious (too hard, nothing finished). Aim for the middle.

A good topic usually...

  • Has a specific, answerable question — not "let's look at climate data" but "how do European heating patterns shift after a heatwave?"
  • Uses real, accessible data — something you can actually get, not something hypothetical.
  • Fits in the time available — a few weeks of work for 3–4 people, not a master's thesis.
  • Has a clear output — a visualisation, a model comparison, a set of statistics, a tool. Something you can point to at the end.
  • Is interesting to your group — weeks of work on a boring question is miserable.

Example topic directions

Not a prescriptive list — just to give you a sense of scope. Your actual topic can come from anywhere.

Data analysis on a public dataset

Explore a question using OECD, Eurostat, World Bank, or a Kaggle dataset. Produce clear visualisations and descriptive findings.

Web scraping + exploratory analysis

Scrape data from a public website, clean it, analyse it. Great for topics where no pre-made dataset exists.

Reproducing a published paper

Pick a paper with public data, try to reproduce its main result, document where you succeeded or struggled.

Comparison across countries / time / groups

Compare something measurable across a dimension — countries, years, demographics — using open data.

Model comparison

Pick a dataset and compare several modelling approaches on it. Focus on the methodology, not on winning a benchmark.

A small tool or dashboard

Build a Shiny app, Streamlit dashboard, or Quarto document that lets someone explore a dataset interactively.

§ 04

Scoping the work

The single most common failure mode in student projects is overscoping — committing to something you cannot possibly finish, then panicking at the end. A few heuristics to help you calibrate.

Signs of good scope

  • You could describe the minimum viable version of your project in one sentence, and it still feels meaningful.
  • You have clear stages: data collection, cleaning, analysis, visualisation — and rough time estimates for each.
  • You know what you'd cut first if you ran out of time.

Signs of bad scope

  • You plan to "try different approaches and see which works" without knowing what "work" means.
  • Your data source requires access you don't yet have.
  • Your method depends on techniques no one in the group has used before and nobody has allocated time to learn.
  • Your plan would require a full-time researcher months of work.
The honest test

If you can imagine a weekend where the whole group sits down and produces a working first version — even a rough one — your scope is probably right. If even a rough first version seems implausible, cut the scope.

§ 05

The repository — what to build

Your one deliverable is a public Git repository. Everything you produce — code, documentation, rendered outputs — lives here.

Minimum structure

This is a suggested layout. Adapt as needed, but include something functionally equivalent.

your-project/
├── README.md — project description, setup, how to run
├── data/ — raw data, or a script that fetches it
├── R/ or src/ — your code, organised by purpose
├── output/ — rendered plots, tables, reports
├── renv.lock or requirements.txt — pinned dependencies
├── .gitignore — ignore caches, credentials, rendered binaries
└── analysis.qmd or main.py — the main entry point

README.md — the single most important file

A classmate or instructor should be able to open your README and immediately understand: what this project is, how to set it up, how to run it, and what it produces. Aim for something structured like:

# Project title

## Authors
Names, course year, group number.

## What this project does
Two or three sentences. Plain language.

## Requirements
- R >= 4.3, or Python >= 3.10
- Any system-level dependencies

## Setup
Exact commands to install dependencies and prepare the environment.

## How to run
Exact commands to reproduce the analysis from start to finish.

## Expected output
What gets produced, where, and roughly how long it takes.

## Data
Where the data comes from, whether it's in the repo or fetched at runtime.

## Repository structure
Brief description of each main folder.

Good practices expected throughout

  1. Commits are meaningful — "fix bug" is bad; "fix off-by-one error in regression output" is good. We will look at commit history.
  2. Code is documented — comments explain why, not just what. Functions have brief docstrings.
  3. No absolute paths — nothing like /Users/your-name/.... Use relative paths or the here package / pathlib.
  4. No secrets committed — no API keys, passwords, or personal tokens in the repo. Use environment variables or .env files that are gitignored.
  5. Multiple contributors — Git should show commits from all group members, not just one.
§ 06

Reproducibility expectations

This is a reproducible research course. Reproducibility is not a bonus; it is the substance of what we are grading.

What "reproducible" means here

Someone with a fresh machine, no prior contact with your group, and only your README in front of them should be able to:

  1. Clone the repository git clone ... — no access requests, no special permissions needed.
  2. Set up the environment Following your README. Tools like renv::restore() or pip install -r requirements.txt should just work.
  3. Run the analysis end-to-end One command, or a short sequence of commands. Not "first open this script, comment out line 47, then..."
  4. Get the same output The figures, tables, and results in output/ should match what your analysis regenerates.

Tools we expect you to use

  • Git for version control — covered in class.
  • renv (R) or venv / conda (Python) for environment management — covered in class.
  • Quarto or R Markdown if you produce a rendered report.
  • Docker is optional — a nice-to-have, not required.
Grading philosophy

We grade reproducibility on best-effort and documentation quality, not tooling purity. You do not need Docker. You do not need perfect setup on every possible operating system. What you do need: a README someone can follow, pinned dependencies, working code, and no hidden assumptions about your own machine.

§ 07

The final presentation

At the end of the semester, each group presents their project live. Two groups present per 90-minute lecture. Your slot is 40 minutes and includes a short talk, a live clone-and-run demo, and Q&A.

The distinctive part: during your slot, you will clone your own repository fresh on the spot and run it in front of the class. This tests what we have been teaching all semester — that a well-built project runs from scratch on any machine.

All logistics, timings, role assignments, and what to do when the demo breaks are covered in the separate presentation guidelines document:
